# Build Mode: drawn buildings, scaling stats, and Labor

Filed 2026-07-21. Replaces fixed tetromino footprints with click-and-drag drawing, makes every stat a
function of the drawn area, and introduces Labor as the construction throughput resource. Star Ruler 2
component sizing, on squares.

Supersedes the placement half of `2026-07-20-buildings-cities-society.md`; the roster, tech gating,
species and city-stage work in that doc still stands and should be built on top of this.

---

## 1. The core change

Today a building is a fixed `Vector2Int[] shape` — an L-tromino, a 2x3, a 3x3 — placed whole. That
shape is baked into `SurfaceBuildingInfo` and every downstream system (footprint, occupancy, power
coverage, adjacency, rendering) reads it.

After this change a building is **a set of cells the player drew**, and every number it produces or
consumes is derived from how many cells that is.

`PlacedBuilding` gains `List<Vector2Int> cells` (relative to origin) and `shape` becomes the *default
suggestion* for a fresh drag rather than the definition. `SurfaceBuildingInfo` keeps a `minCells` and
`maxCells` per type — a Power Node is 1-2, a Spaceport 6-25 — because "draw a 400-tile farm" should be
refused by the building, not by the wallet alone.

**Serialization:** `PlacedBuilding` is serialized as itself by `JsonUtility`, which cannot handle a
`List<Vector2Int>` field cleanly. Store as two parallel `List<int>` (`cellX`, `cellY`) — flat, ordinal,
JsonUtility-safe, and matching the existing `OreCellDTO` precedent.

---

## 2. Scaling: what each extra cell costs and gives

One rule, applied per building type with its own coefficients. For a building of **N** cells:

| Quantity | Formula | Default |
|---|---|---|
| Metal / Energy cost | `base * N * (1 + growth)^(N-1)` | growth = 0.05 |
| Build time | `baseTime * N * (1 + timeGrowth)^(N-1)` | timeGrowth = 0.05 |
| Upkeep (power draw) | `baseDraw * N * (1 + upkeepGrowth)^(N-1)` | upkeepGrowth = 0.05 |
| Output | `baseOutput * N * (1 + outputGrowth)^(N-1)` | outputGrowth = 0.10 |
| Labor | `laborPerCell * N` | 1.0 |

Output grows *faster* per cell than cost, which is the whole point — a mega-farm is genuinely better
per tile than four small ones. Cost, time and upkeep all compound at 5% while output compounds at 10%,
so the brake is not efficiency but **the up-front wall**: a 20-cell farm costs `20 * 1.05^19 ≈ 50x` a
single cell for `20 * 1.10^19 ≈ 122x` the yield. Affordable only late, and it locks up Labor for a
long time while it builds.

These four coefficients live on `SurfaceBuildingInfo` so a Reactor can compound harder than a Housing
Block. All of them must be modifiable by tech (`TechEffects` already has `BuildCostMult` /
`BuildTimeMult` — add `LaborMult` and `UpkeepMult`).

**Balance risk, stated plainly:** exponential-in-N is very sharp. At growth 0.05 a 40-cell building
costs 280x a single cell. `maxCells` per type is what keeps this bounded, and it needs to be set
deliberately rather than left generous.

## 3. Labor

A throughput pool, modelled directly on the shipyard's existing `BuildPower` — the same shape, so the
two read as siblings rather than two inventions.

**Generated by** (adds to the max pool):
| Building | Labor |
|---|---|
| Planetary Capitol | 2, +1 per upgrade level |
| Colony Ship Base | 1 |
| Storage Depot | 1 per cell |
| City block | 0.5 each |
| Town | 0.25 | 
| Settlement | 0.1 |

**Consumed by** every build in progress: `laborPerCell * N`, held for the duration and released on
completion, cancellation or pause. This is what makes Labor a *rate* rather than a stockpile — you
never spend it, you occupy it.

**Under-supply does not block.** If the queue wants more Labor than exists, the shortfall stretches
the build rather than refusing it:

```
effectiveTime = buildTime * (1 + shortfall / required)
```

so a project with half the Labor it wants takes 1.5x as long, not forever. When another project
finishes and frees Labor, the queue re-allocates and in-flight builds speed up. **Never a hard stop** —
the same reasoning as `PowerGrid.UnpoweredFactor`: a build that silently never progresses is an
undiagnosable bug for the player.

**Allocation order:** queue order, first-come. A build already under way keeps its Labor when a new one
is queued behind it — otherwise starting a second project would slow the first, which reads as a
punishment for planning ahead.

**Display:** a header at the top of the Build tab, mirroring the shipyard's build-power readout —
`Labor: 7 free of 12  (5 in use across 2 projects)` — plus the build queue itself, with per-project
progress bars, pause and cancel.

## 4. Build times

Currently surface buildings appear instantly; only `buildTime` exists as an unused field. This adds the
whole missing layer:

- `SurfaceBuildQueue` — per-body list of `SurfaceBuildOrder { type, cells, origin, rotation, elapsed, laborHeld, paused }`.
- Ticks on scaled time (like every other in-game timer), completing into a real `PlacedBuilding`.
- A **ghost** rendering of in-progress buildings on the map — outlined, semi-transparent, with a
  progress fill — so a world under construction reads as such.
- Cancel refunds the same 60% `Demolish` already uses; pause releases Labor.

## 5. Drawing

**Interaction:** pick a building type, then L-click-drag across the grid to paint cells. Release opens
the existing confirmation window, now showing the derived numbers for what was actually drawn — cost,
build time, Labor, upkeep, output — with Build and Cancel. Cancel discards the drawing and you draw
again; the type stays in hand.

**Rules while drawing:**
- Cells must be orthogonally connected (no diagonal-only links, no islands).
- Each cell must pass the existing per-cell checks: in bounds, buildable terrain, unoccupied, and
  the type's `minIndex` against `SurfaceIndex`.
- Live feedback: valid cells paint in the building's colour, invalid ones in red and are simply not
  added. A running cost/Labor readout follows the cursor.
- Painting over an already-painted cell in the same drag removes it (drag-to-erase), so a misdrawn
  edge doesn't mean starting over.
- Hard stop at `maxCells`, with the readout saying why.

**Where this lives:** `PlanetViewWindow` already owns `selected`, `CarryingPowerPiece`, `ghostLayer`,
`ScreenToCellIn` and the placement path — the drawing state belongs there, and `SurfaceBuildManager`
gains `CanPlaceCells(body, type, cells, out why)` alongside the existing `CanPlace`.

## 6. What this breaks, and what has to move with it

Every one of these reads `info.shape` today and must move to `PlacedBuilding.cells`:

- `SurfaceBuildingDatabase.Footprint(type,x,y,rot)` and `Footprint(PlacedBuilding)` — the two
  chokepoints; most callers go through them, which limits the blast radius.
- `SurfaceBuildManager.CanPlace` / `Place` / `ForcePlace` / `Occupied` / `Demolish` / `FindSpot`
- `PowerGrid.CoverageOf` — coverage is measured from *every* footprint cell, so a drawn shape changes
  grid topology. This is a feature (a long thin reactor spans further) but it is also where the
  performance risk is: coverage is O(cells x radius²).
- `CityGrowth.TrySpawn` / `TryPromote` — grown settlements should get generated blob shapes rather
  than fixed tetrominoes, which is a nice win.
- `GameStateSerializer` load path — `RemoveAll` on unknown types, `NormalizeAfterLoad`, and the
  re-stamping of `tile.occupied` from footprints.
- The Build tab's card list, the ghost preview, and `PlanetViewWindow.DoPlace`.

**Old saves:** a `PlacedBuilding` with an empty `cells` list falls back to `info.shape` in
`NormalizeAfterLoad`. That keeps every existing save valid and is why `shape` stays on the info rather
than being deleted.

## 7. Slices

**Slice 1 — cells as data.** `PlacedBuilding.cells` + serialization + `Footprint` reading from it, with
`shape` as the fallback. No UI change, no drawing: existing buildings place exactly as before but are
*stored* as drawn shapes. Everything downstream keeps working. This is the risky structural slice and
it ships on its own so it can be tested in isolation.

**Slice 2 — the scaling model.** The five formulas, the per-type coefficients, `minCells`/`maxCells`,
and a stats readout. Still placing fixed shapes — but now a 4-cell farm computes its numbers from 4.

**Slice 3 — Labor.** `Labor` static (modelled on `BuildPower`), generation from the listed buildings,
the max pool, and the Build-tab header. Nothing consumes it yet.

**Slice 4 — build times and the queue.** `SurfaceBuildQueue`, ghost rendering, Labor consumption and
the shortfall stretch, pause/cancel/refund, and the queue UI under the Labor header.

**Slice 5 — drawing.** Click-drag painting, connectivity and validity rules, drag-to-erase, the live
readout, and the confirmation window wired to the drawn cells.

**Slice 6 — the rest.** `CityGrowth` blob shapes, `FindSpot` for drawn buildings, `PowerGrid` coverage
performance pass, and a balance pass across the whole economy with scaling live.

## 8. Open questions for Jacob

1. **Can a drawn building be non-rectangular?** L-shapes, rings, snakes? Assumed yes (that is the
   point), but it makes `PowerGrid` coverage and the ghost renderer meaningfully harder.
2. **Does rotation still exist** once shapes are drawn? Assumed no — you draw it the way you want it.
3. **`maxCells` per type** — needs your numbers, or I set them and you tune.
4. **Should Labor be per-world or empire-wide?** This doc assumes **per-world** (it is generated by
   buildings on that world), which means a young colony builds slowly until it has a capitol and depots.
   Empire-wide would let a developed homeworld carry a new colony — a very different game.
