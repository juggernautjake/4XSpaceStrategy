# Build Mode overhaul — drawn footprints, Labor, and a build queue

**Source:** Jacob, 2026-07-21. "Star Ruler 2 ship components, only with squares instead of hexagons."
**Status:** Planned, not started.

## What changes

Buildings stop having fixed tetromino shapes. You pick a type, then **click-and-drag to colour in the
tiles** you want it to occupy. Everything about that building — cost, build time, upkeep, output, and
the Labor it ties up — scales with how many tiles you drew.

The confirmation step stays: draw, look at what it will cost, then Build or discard and redraw.

## The three formulas

For a building drawn across **N** tiles, where block *i* is 1-indexed:

| | Per-block | Total for N |
|---|---|---|
| **Cost, build time, upkeep** | `base × (1 + 0.05·(i−1))` | `base × (N + 0.05·N(N−1)/2)` |
| **Output** | `base × (1 + 0.10·(i−1))` | `base × (N + 0.10·N(N−1)/2)` |
| **Labor** | `laborPerTile` (default 1) | `laborPerTile × N` |

So a 3-tile farm costs **3.15×** and produces **3.3×**; a 4-tile farm costs **4.3×** and produces
**4.6×**. Mega-farms are possible and deliberately expensive — output rises faster than cost, but
build time and upkeep rise with cost, and Labor rises linearly, so the real brake is that a big
building monopolises the workforce for a long time.

All six constants live in one place so they can be re-tuned by eye.

## Labor

Modelled on `FacilityPower.BuildPower`, which already does exactly this job for shipyards — a pool
that projects draw from while they run and give back when they finish.

**Generated by:**
| Building | Labor |
|---|---|
| Planetary Capitol | 2 at level 1, +1 per upgrade |
| City blocks / housing | 0.5 each |
| Storage Depot | 1 **per grid square** |
| Colony Ship Base | (government — same treatment as the capitol) |

**Consumed:** every build project holds `laborPerTile × tiles` from the pool from the moment Build is
confirmed until it completes, is cancelled, or is paused. Higher-tier buildings (reactors) cost more
than 1 per tile.

**Starved projects still run, just slower.** If the queue wants more Labor than the world has, the
shortfall stretches build time rather than blocking the project — one missing Labor is a fixed
percentage penalty on the remaining work. When something finishes and frees Labor up, the next project
in the queue picks it up automatically.

Displayed at the top of the Build tab exactly as the shipyard shows Build Power: **used / max**, with
the queue underneath.

## Slices

### 1. Free-form footprints — the data model
- [ ] `PlacedBuilding` carries its OWN cells rather than deriving them from `SurfaceBuildingInfo.shape`
- [ ] Save/load round-trips them (flat parallel int lists, matching `BodyDTO`'s existing discipline —
      JsonUtility will not nest a `List<Vector2Int>` safely)
- [ ] `SurfaceBuildManager.CellsOf/Footprint/CanPlace/Place/Occupied` read the stored cells
- [ ] **Existing saves keep working:** a `PlacedBuilding` with no stored cells falls back to its
      type's authored shape, so worlds built before this still stand

### 2. Drawing
- [ ] Click-drag paints cells; release opens the confirmation window
- [ ] Live validity while dragging (green/red per cell, same language the ghost uses today)
- [ ] Contiguity rule — a building is one connected blob, not scattered tiles
- [ ] Cancel and redraw without spending anything
- [ ] Replaces the tetromino ghost + rotation for drawn types

### 3. Scaled economics
- [ ] The two curves above, in one tunable place
- [ ] `EfficiencyAt` still averages the driving index across the drawn cells (unchanged rule, new shape)
- [ ] Upkeep/power draw scale with tiles — `PowerGrid` reads the scaled draw

### 4. Labor
- [ ] `SurfaceLabor` — max from generators, used by live projects, mirroring `FacilityPower`
- [ ] Per-building `laborPerTile`, modifiable by research (so tech can move it later)
- [ ] Shortfall stretches build time; freed Labor flows to the next queued project

### 5. Build times and the queue
- [ ] Every building gets a real build time (none of them tick today)
- [ ] Per-world build queue with progress, pause, cancel-with-refund, drag-reorder — the same
      affordances `ShipyardWindow`'s stocks already have, because the player already knows them
- [ ] Queue + Labor readout at the top of the Build tab

### 6. Later: ship and station parts
Explicitly deferred by the request. The drawing and the scaling curves are the reusable half; hull
layout is its own problem.

## Notes

- The **grown** settlement types (`Settlement`, `Town`, `City` — placed by `CityGrowth`, not by the
  player) must keep working throughout. They are placed programmatically and have no draw step.
- `uniquePerWorld` and `allowMultiple` still apply — drawing does not change how many of a thing a
  world may have.
- Nothing here is compiled by the agent that writes it (no Unity in that environment), so each slice
  is reviewed by independent agents instead. Build before playing.
