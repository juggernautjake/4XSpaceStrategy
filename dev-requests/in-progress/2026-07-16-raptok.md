# Requests from Raptok — 2026-07-16

**Source files:** `Planet View UI Update.md`
**Status:** In progress

## What they asked for

Two follow-on features for the Planet View built in the previous (2026-07-15) request:

**1. A tile hover-info window.** While hovering the surface map, show a small, semi-transparent
window near the cursor with the tile's biome type, what's on it, and the planet's temperature — the
tile-type text coloured like the tile itself, the temperature coloured on a gradient (white = coldest
of all planets → ice blue → yellow-orange → red = hottest of all planets), and the window's bottom-left
corner anchored to the mouse with a slight rightward offset so the cursor never covers it. Fetched from
existing data, no new assets. The request also asks, at length, for a believable per-planet temperature
if one doesn't already exist, referencing terraforming so hot/cold planet types read as consistently
hot/cold on that gradient.

**2. A moon-map overhaul.** Zoom/pan on open moon maps (the host map already has both). Move the moon
tabs from the bottom of the map to a vertical stack anchored inside the map's top-left corner. Turn the
tabs from name-labelled buttons into small square thumbnails of the moon. Add a hover window on each tab
(name + moon description) anchored the same way as the tile window.

### Ambiguities I resolved

- **"Average temperature" on a tile hover:** the surrounding paragraphs describe a *global*, un-renormalized
  gradient where a planet's *type* decides which end of the scale it lives on — that only makes sense read
  per-tile (so a hot world's tiles cluster toward red even at their local coldest, a cold world's toward
  white/blue even at their hottest). I read "average" as loose wording for "the tile's temperature," not a
  literal average. I also added a genuine body-level average (see below) and surfaced it as a new stat in
  the Survey tab's Climate card, since the request separately and explicitly asks for a "planet Temperature
  setting for generation" — that request stands on its own regardless of what the tooltip shows.
- **"Resources present" on a tile:** read as the tile's own ore deposit (`TerrainTile.ore`), not the body's
  aggregate Metal/Energy/Water pool — it's the one per-tile resource fact the game already tracks, and
  matches "hovering over a tile" rather than "hovering over a planet."
- **No existing "planet Temperature" field:** there wasn't one, only `terrainParams.heat` (a 0.45–1.85
  multiplier already driven by distance-from-star, already the thing the terrain classifier and
  `TerraformVisuals` blend during terraforming). Rather than add a second, disconnected temperature that
  could drift from it, I built a real Celsius reading **on top of** that existing multiplier
  (`PlanetTemperature`, derived not stored) — so it moves correctly as a world terraforms, matches "look at
  terraforming logic," and needed no new generation code or save data.
- **"Type of moon":** `CelestialBodyType` has no moon sub-types (icy/volcanic/etc.) — a moon is just
  `Moon`. I derive the "type" description from the moon's own generated surface (its dominant terrain,
  already fetched from `moon.surface.tiles`) rather than inventing a new taxonomy or asset.
- **Zoom/pan on moon maps:** built as an independent, additive per-moon zoom/pan (its own state per open
  moon), rather than folding moon maps into the host's existing zoom/pan/pan-clamp/confirm-panel machinery.
  That machinery is deeply wired to placement (ghosts, confirm panel, selection markers) which moon maps
  don't have; a shared refactor would touch far more of this 3200-line file for no behavioural gain.

## Slices

### 1. Planet Temperature model + Climate stat
- [x] A new `PlanetTemperature` helper derives a believable Celsius reading from a body's existing
      `terrainParams.heat` (plus a small latitude swing per tile and a type nudge: volcanic worlds run hot,
      ice/gas-giant worlds run cold), so it updates correctly as a world terraforms and needs no new
      generation code or save data.
- [x] A global (not per-planet-renormalized) white → ice-blue → yellow-orange → red colour gradient over a
      fixed temperature range, so a planet's *type* decides which end of the scale its tiles land on.
- [x] The Survey tab's Climate card shows the body's average temperature, colour-graded the same way —
      the explicit "add a planet Temperature setting" ask, surfaced somewhere a player can already see it.

**Notes:** Foundation for slice 2; no UI beyond the one new stat line. Reviewed by 2 subagents (compile +
formula sanity, working the actual Celsius numbers by hand at the heat extremes for every body type).
Compile pass was clean. The sanity pass confirmed Volcanic/Ice worlds keep to the correct half of the
gradient with a wide margin (~40-50C from the wrong stop) and the ±15C per-tile latitude swing never
crosses a stop boundary, and found one small calibration miss — the white anchor was 10C colder than any
world could ever actually average — which I fixed (-140C to -130C). It also flagged that BarrenPlanet,
having no type nudge, can read as only mildly warm/cool right at its distance-band edges rather than
strongly saturated; that's a soft cosmetic edge, consistent with the request's own "will likely never"
(not "never") wording, so I left it as-is rather than adding a Barren-specific modifier.

### 2. Tile hover-info window
- [x] Hovering a tile on the host map shows a small, semi-transparent floating window: the tile's biome
      name (coloured like the tile), its ore if one is discovered there, and its temperature (coloured on
      the slice-1 gradient) — anchored by its bottom-left corner to the mouse with a slight rightward
      offset, with a text shadow so it stays legible over any tile colour underneath it.
- [x] Works across every Planet View tab, not just Survey, since the request says "while hovering ... the
      terrain tiles" without restricting it to one tab.

**Notes:** New `MapHoverPanel` (self-creating, like `TooltipManager`, but semi-transparent and anchored by
its bottom-left corner rather than its top-left) shared with slice 3b below rather than building two
one-off windows. Reviewed by 4 subagents total across two passes (compile, clean; then logic, which found
two real bugs and a verification pass that confirmed both fixes). The logic pass found: (1) closing the
window via the HUD's toggle button — not the title-bar X, not click-away — never hid the panel, since
`Update()` (the only thing driving the per-frame hover poll) stops running the instant the window goes
inactive; fixed by hiding it in `Toggle()`'s close branch too. (2) The panel's canvas sat above the zoom
bar, moon-tab strip and the Build confirm dialog, so hovering near those covered their own buttons/text;
fixed by suppressing the tooltip over those three rects, the same way `PollMapPan` already excludes them
from drag-to-pan. A verify pass confirmed both fixes are correct and didn't reopen anything.

### 3a. Moon tabs: top-left vertical stack of thumbnails
- [x] Moon tabs move from the bottom-centre row to a vertical stack anchored inside the map viewport's
      top-left corner (closest moon still first/topmost).
- [x] Each tab is a small square (no taller than the current tab height) showing a downscaled image of
      that moon instead of its name as text.

**Notes:** Square buttons sized to the OLD tab height (18px, `MoonTabSize`) so they're never larger than
before; each shows a `RawImage` built from `SurfaceTextureRenderer.BuildGrid` — the same renderer the open
moon maps already call, just at button size — tracked in a new `moonTabThumbTextures` list and freed on
every rebuild/destroy alongside the existing moon-map textures. Reviewed by 1 subagent (compile + a check
that nothing downstream still expected the old horizontal layout or a name-label child); clean.

### 3b. Moon tab hover window
- [ ] Hovering a moon tab shows the slice-2 `MapHoverPanel`, anchored the same way, with the moon's name
      and a short description of what kind of moon it is (derived from its own generated terrain).

### 4. Zoom and pan on open moon maps
- [ ] Each currently-open moon map can be scrolled to zoom and dragged to pan, independently of the host
      map and of each other, the same way the host map already works.

**Notes:** Slices 3a/3b/4 are the moon-map overhaul; 3a and 3b touch the same tab code so are ordered
together, 4 is independent of both and is the largest/riskiest of the four (a new self-contained
per-moon zoom/pan rather than a shared refactor — see the ambiguity note above). If the run's budget
runs out, these are what's left `- [ ]` for the next pass — slices 1 and 2 (the tile window, the part of
the request with by far the most explanation) are the ones that had to land regardless.
