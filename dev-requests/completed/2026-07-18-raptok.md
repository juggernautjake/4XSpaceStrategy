# Requests from Raptok — 2026-07-18

**Source files:** `Planet-Moon Generation, Changing of Sandbox Terrain Sliders.md`
**Status:** Complete

## What they asked for

Two linked asks in one file:

1. **Terrain Sandbox slider rework** — replace the Elevation slider with a "Water Level" slider that
   reads as actual sea coverage (full-right = whole surface underwater, pulling it down reveals
   islands); replace Moisture with a "BioSphere" slider that only produces plant-life tiles when the
   world is actually habitable (water + warm-enough temperature; atmosphere is named too, see below);
   widen the Temperature slider so it reaches truly extreme cold/hot; drop the Mountains slider for
   now. Cold worlds should render their water as ice/snow, and warming an ice world should melt it into
   real ocean tiles. Terraforming that adds/removes water should move the Water Level slider live.
   Microbial Seeding should be the only way a barren world gains its first BioSphere, gated on
   water/temperature (and atmosphere, which doesn't exist as an attribute yet — see below), warning
   before failure.
2. **"Advanced Planet Generation"** — a full redesign where planet/moon *type* is no longer picked
   directly, but emerges from new first-class attributes (Size, Temperature, Water Level, Tectonics,
   Atmosphere Thickness, BioSphere), all of which stay editable after generation via terraforming; plus
   a wholly new Tectonics system (continent plates, fault lines, a Survey overlay with directional
   arrows, mountain/volcano placement along faults, earthquake events, mineral-deposit interaction); plus
   reworking moon generation onto the same attribute system so moons stop being one uniform "airless"
   palette.

**Scale call:** (2) is a multi-system architectural rewrite — new persistent attributes threaded through
generation, terraforming, and three separate UI surfaces, plus an entirely new overlay/event system for
Tectonics. That's not a slice, it's a project. This run scopes and builds what's tractable from (1), which
is concrete and self-contained, and records (2) as explicit follow-up objectives rather than attempting a
rushed version of a system this size. Per the job's own rules: shipping a handful of solid slices beats a
sprawling half-built rewrite that's harder to pick up cleanly next time.

**Ambiguities / assumptions I made:**
- *"Elevation slider kept for Sandbox/Dev Mode but not available in player mode"* — today the entire
  Terrain tab is already Dev-Mode-gated (`PlanetViewWindow.TabAvailable`, `Tab.Terrain` requires
  `GameMode.DevMode`); there's no player-facing terrain editor at all yet. I read this as describing a
  *future* player-facing water view that doesn't exist yet, not a regression to fix now. I left the
  Terrain tab's Dev-Mode-only gating untouched and just reworked the sliders that live inside it.
- *"BioSphere cannot increase unless... atmosphere must exist"* — there is no `atmosphereThickness` (or
  any atmosphere) attribute on `CelestialBody` today; "atmosphere" only exists as a terraform-diagnosis
  problem computed on the fly, not a stored value. I gated BioSphere/Microbial-Seeding on Water Level and
  Temperature only, and called out the missing atmosphere gate explicitly rather than faking an atmosphere
  check against nothing. A real atmosphere attribute is squarely part of the (2) Advanced Generation
  rework.
- **Water Level implementation**: rather than inventing a second, disconnected "water level" number, I
  repurposed the existing `terrainParams.elevation` amplitude (which every biome classifier already
  thresholds land-vs-water against, and which terraforming's water projects already push up/down) and
  just changed how the sandbox slider displays/edits it — inverted, so higher slider = lower elevation
  amplitude = more of the map falls below the water threshold. This means "terraforming already moves the
  Water Level slider live" falls out for free: terraform code was already nudging this same field.

## Slices

### 1. Terrain Sandbox panel — Water Level, wider Temperature range, drop Mountains
- [x] Elevation slider replaced with a "Water Level" slider (0 = driest/most land, 1 = fully
      water/ice-covered), built on the existing `terrainParams.elevation` field with an inverted
      display mapping so the UI direction matches the request (full right = whole surface covered).
- [x] Temperature slider range widened toward the cold extreme so a genuinely frozen world is reachable
      (the hot end was already saturating against the existing `clamp01` in `SampleNormalized` well
      before the old max, so I noted that instead of pretending a wider top end does more than it does —
      review found the hot-end claim was slightly overstated for the polar band, comment corrected).
- [x] Mountains slider removed from the sandbox panel for now; `ridge` is still generated/terraform-driven,
      just no longer directly sandbox-editable.

**Notes:** Pure UI/param-mapping change in `PlanetViewWindow.cs`, no new data fields. Lowest risk, and
slice 2/3 build on the same panel, so doing this first establishes the pattern.

### 2. Ice/Cold worlds — water freezes and thaws with temperature
- [x] `Ice()` biome classifier (`PlanetTerrainGenerator.cs`) becomes temperature-aware like `Terran()`
      already is: below freezing, low-elevation ("water level") tiles render as FrozenSea/Ice/Snow;
      warmed above freezing, those same tiles melt into Ocean/Lake. A maxed Water Level slider on an Ice
      world is now an ice-covered planet while cold and an ocean world once warmed through terraforming,
      matching the request's melt/refreeze description. (First review pass caught the melted fallback
      over-reaching into Ocean for a huge mid-elevation band; fixed to melt into Tundra instead so only
      the genuinely low-elevation band becomes open water, re-verified after the fix.)

**Notes:** Touches only `Ice()`'s signature and body; `Classify()`'s dispatch already passes `temp` to
every other classifier so this is a small, contained change reusing an existing thresholds pattern.

### 3. BioSphere attribute, slider, and Microbial Seeding gating
- [x] `CelestialBody.biosphereActive` (bool) added — true at generation only for worlds that start out
      warm and wet enough to sustain plant life; false otherwise (barren/volcanic/ice/gas/moons/asteroids
      by default, and rocky/ocean worlds outside the liquid-water band).
- [x] Moisture slider renamed/restructured to "BioSphere" in the sandbox; plant-life tile density is
      floored near-zero whenever `biosphereActive` is false or Temperature/Water Level sit outside the
      liquid-water range, with a note explaining the cap instead of silently ignoring the drag. Review
      caught that my first pass only capped the sandbox UI's displayed slider — the actual generator
      (`Terran()`) never checked `biosphereActive` at all, so ordinary terraforming could still grow
      jungle tiles on a world that never had Microbial Seeding. Fixed by gating moisture once, upstream,
      inside `PlanetTerrainGenerator.SampleNormalized` (before both the classifier and the exposed
      `Sample.moisture`, so terrain and the Fertile/Solar survey overlays can't disagree with each other).
- [x] Microbial Seeding terraform project gains a pre-activation viability check (water level +
      temperature — atmosphere deliberately excluded, see assumptions above) that reports a failure
      warning when conditions aren't met, and sets `biosphereActive = true` on success so a previously
      barren world can start generating plant tiles.

**Notes:** The only slice that adds a persistent field; kept the gate to the two attributes that actually
exist (water, temperature) rather than stubbing a fake atmosphere check. Five review rounds total:
(1) found the sandbox-only gating gap, (2) a moisture/Sample-field consistency bug in the first fix
attempt — both fixed and re-verified; (3) a holistic pass across all 6 touched files then found
`biosphereActive` was being set AFTER a world's first surface bake in three separate generation paths
(main loop, the "guaranteed habitable world" backstop, and the home world), and that it wasn't
persisted through save/load at all (would silently reset to false — and visibly de-green a terraformed
world — on every load); (4) verified those follow-up fixes (3 generation-order fixes + BodyDTO/
GameStateSerializer persistence) end to end. Review agents on this slice: 5.

### 4a. Moon/asteroid terrain reacts to orbital temperature
- [x] `Airless()` classifier (moons + asteroids) now takes `temp`: the low-elevation band that used to
      be unconditionally `Ice` now renders `CrackedGround` once the body is warmed above the same
      freezing threshold `Terran`/`Ice` use, so `BiasHeat`'s per-moon orbital-distance heat value (which
      was already computed and stored but silently had zero effect on the map) actually shows. A moon
      close to a hot star and one in deep cold now genuinely look different.

**Notes:** A small, self-contained slice off the front of §4b below — full moon-TYPE variety (ocean/
volcanic/ice moon subtypes) still needs the attribute-driven generation rewrite, but this one-line gap
(a value computed and stored, then never read) was cheap and safe to close on its own. Reviewed by 1
agent; found that `OreGenerator.TerrainAffinity` has no explicit case for `CrackedGround` (falls to the
0.02 default rather than `Ice`'s 0.05) — this predates my change (`Volcanic()` already produces
`CrackedGround`) and is a pre-existing ore-tuning gap, not something this fix introduced, so left as-is
rather than expanding scope.

### 4c. Atmosphere Thickness attribute
- [x] `CelestialBody.atmosphereThickness` (0..1) added — a new `AtmosphereRules.ForBody(type, surfaceSize)`
      computes it at every generation/retype site from Size and Type: gas giants max, volcanic worlds
      thicker than a same-size rock (outgassing), asteroids always none, moons none unless "large"
      (thin above that), other planet types scale thin-to-moderate with size. Persisted through save/
      load (`BodyDTO`/`GameStateSerializer`, with a backfill for saves from before this field existed).
- [x] Wired into the Solar and Wind Survey indices (`SurfaceIndex.cs`): thicker atmosphere now lowers
      Solar (blocks/reflects light) and raises Wind (air to move at all — asteroids and small moons now
      correctly read zero Wind); thin/no atmosphere is the reverse, per the request.
- [x] Wired into `PlanetTemperature` as a greenhouse term (extra warmth independent of stellar distance,
      the Venus-vs-Mercury effect the request calls out by name), and into `BiosphereRules` as the
      previously-deferred "atmosphere must exist" gate from slice 3 — `MicrobialSeedingWarning`/
      `WhyCapped` now mention a too-thin atmosphere as a real failure reason.

**Notes:** Split off the front of the original first §4b bullet — this is the one sub-piece of "promote
Size/Temperature/Water Level/Tectonics/Atmosphere Thickness/BioSphere to first-class attributes" that's
genuinely self-contained (an additive attribute + two index formulas + one gate), unlike Tectonics or
the type-selection rewrite which need the rest of §4b's architecture first. Reviewed by 2 agents: the
first pass found a real bug (`TerraformManager.Reshape` — reachable from World Remodelling/Hydrosphere
Venting/Crustal Sequestration — changed a body's TYPE without ever recomputing `atmosphereThickness`,
so a remodelled world kept stale air forever) and a stale doc comment, both fixed; the second verified
the fix and confirmed no other retype site has the same gap. Known limitation: `MinAtmosphere = 0.12f`
in `BiosphereRules` is currently unreachable for any freshly-generated Rocky/Ocean planet (the type
baseline of 0.15 already clears it) — the gate only matters for a former small moon remodelled into a
habitable type, not for a planet ever being "too thin" on its own. Tuning that baseline lower to make
genuinely-thin rocky worlds possible would need real playtesting to calibrate against, which isn't
available here; documented rather than guessed at.

### 4d. Tectonics attribute (presence only)
- [x] `CelestialBody.hasTectonics` (bool) added — rolled once at generation (and re-rolled on any later
      `TerraformManager.Reshape` type change, same lesson as §4c) via a new `TectonicsRules.Roll(type,
      surfaceSize)`: never for gas giants/asteroids, only for "large" moons at a modest chance, and for
      terrestrial planets a chance centred close to the request's own "~1/3 of the time" starting guess,
      weighted toward larger worlds. Persisted through save/load (no backfill needed — see notes).
- [x] `TectonicsRules.BoostRidge` gives an active world a real, visible effect today: a one-time mountain-
      noise amplitude boost applied at generation, which every existing biome classifier already reads
      (Mountains/Highlands/Canyon/CrackedGround thresholds), so a tectonically-active world is genuinely
      more rugged without needing any new per-classifier code.
- [x] `OreGenerator.Populate` now boosts ore affinity and tier acceptance across the whole world when
      `hasTectonics` is true ("rare or high quality mineral deposits are more common there" from the
      request) — a whole-world proxy for the Mineral-overlay interaction, since there's no fault-line
      geometry yet to concentrate the bonus AT the boundaries specifically (that part stays in §4b).

**Notes:** This is the presence ATTRIBUTE only — the fault-line geometry, the Survey overlay with
directional arrows, mountain/volcano placement specifically ALONG fault lines, and earthquake events all
need real fault geometry (which plates, which shared edges, which direction each is pushing) that
doesn't exist yet; those remain in §4b below. Reviewed by 3 agents: first pass confirmed the attribute/
wiring compiles and is ordered correctly everywhere, but caught the same `Reshape`-doesn't-recompute-on-
retype bug §4c had already been fixed for one commit earlier — fixed here too, then verified; a third
pass reviewed the ore-affinity change separately and confirmed the probability math and call-site
ordering are correct. Deliberately did NOT add a save-load backfill for old saves (unlike
`atmosphereThickness`): `BoostRidge`'s effect is baked directly into the persisted `ridge` value at
generation time, and `GameStateSerializer` restores terrain params from those stored floats rather than
re-running generation on load — so a pre-existing save reading `hasTectonics=false` has no observable
gameplay effect today (nothing else reads the flag yet). This will need revisiting once a Tectonics
overlay actually reads the flag directly.

### 4e. Water Level genuinely drives type selection in the temperate band
- [x] `RollBodyByTemperature` now rolls a real, independent Water Level value in the temperate
      (habitable) band and uses IT — not a bare anonymous random draw — to decide Ocean vs. Rocky vs.
      Barren; the exact same roll is then fed straight into the body's actual terrain (`terrainParams.
      elevation`, via `ElevationFromWaterLevel`), so type and map agree: a bare-minimum "Ocean" world
      reads as mostly islands, a maxed one is fully drowned, exactly as the request describes the Water
      Level slider. Calibrated so the Ocean/Rocky/Barren proportions (45%/40%/15%) are bit-for-bit
      identical to before — this is a genuine rewire, not a balance change.

**Notes:** This is the ONE piece of "drive type selection from attributes" that's safe to do without
playtesting: it only touches the temperate band, whose type choice was already meant to be about water,
and it's calibrated to reproduce the exact prior proportions so there's no hidden balance shift to
verify by eye. What's still NOT attribute-driven (see §4b below) is Gas Giant/Ice/Asteroid frequency —
those come from Size in the request's vision, and reworking that touches how often each type appears
across the WHOLE galaxy, which is a real balance question only playtesting can answer, not something to
guess at blind. Reviewed by 2 agents: the first did the exact probability arithmetic confirming the
45/40/15 split is preserved and the elevation math produces sane results at both extremes and the
Rocky/Ocean boundary, and separately surfaced a serious, unrelated PRE-EXISTING bug — every top-level
planet generated through the normal path was ending up with ZERO ore, because `OreGenerator.Populate`
ran inside `MakeBody` against a provisional surface that `GenerateSystem` always immediately discards
and rebakes (once `BiasHeat` sets the real climate), with nothing re-populating ore on the surface that
actually ships. Fixed by moving the `Populate` call to after the real, final bake. The second review
verified the fix and found it also incidentally repairs "SpecialResource" POI placement, which depends
on ore already being on the surface it scans — that was silently broken by the same root cause.

### 4b. Advanced Planet Generation — BUILT 2026-07-18 (second session, Opus)

The project deferred by the first session was built out in full across six reviewed slices (see the
commits on `main`, and each box's note for where it landed). This section is why the original run went to
**In progress**; it is now done.

- [x] Moon generation uses the same attribute system as planets — a moon rolls a real world-type
      (Ocean/Rocky/Volcanic/Ice/Barren, or stays airless Moon) from its temperature + a Water Level draw,
      gated by its mass: only a large moon holds the air/water to come out temperate/ocean, a hot orbit
      turns any moon volcanic (Io), a cold one icy (Europa). A large warm wet moon can even start with a
      biosphere. `SolarSystemGenerator.RollMoonType` + a new `AtmosphereRules.ForMoon` (moon-mass air cap
      for ALL surface types, so a small ice/volcanic moon stays airless) + a `PlanetAppearance` thickness
      guard so those airless moons look airless. Home-world moons deliberately stay airless (hand-tuned,
      guaranteed-terraformable starting case). `[commit: "Moon variety"]`
- [x] Tectonics fault-line GEOMETRY — new `TectonicsMap`: Voronoi continental plates derived
      deterministically from the terrain seed (like `SurfaceIndex` — nothing stored, re-derives on load),
      answering at any (u,v) which plate a point is in, how close it is to a fault, and whether that fault
      is convergent (compression → mountains) or a divergent rift. Each plate carries a push direction +
      strength. Uses `System.Random`, so deriving it never perturbs the generation RNG stream.
- [x] Fault lines bias mountain/volcano placement ALONG them — `PlanetTerrainGenerator.SampleNormalized`
      lifts the ridge field near convergent faults (and lowers it at rifts), so ranges and volcanoes
      concentrate on the fault lines the overlay draws; a drowned fault stays sea (Earth's mid-ocean
      ridges). Reordered the Ice classifier so its water line is tested before its Mountains threshold
      (review fix — otherwise a fault raised peaks out of the frozen sea). The boosted ridge also lifts the
      Mineral index at fault zones, so "rare/high-quality deposits cluster near faults" reads on the map
      (on top of §4d's whole-world ore boost for tectonic worlds).
- [x] Tectonics Survey overlay + directional arrows — a "Tectonics" Survey overlay: translucent white
      plate wash, red fault-line cells (1-3 wide, wider where plates meet), and a red push-arrow per plate
      at its centre, sized by push strength. Reads the same `TectonicsMap` as the terrain, so the red lines
      land exactly where the mountains do. Survey-gated; mutually exclusive with the index ramps and the
      power overlay. Built as a bespoke overlay (like the power grid) rather than a `SurfaceIndexKind`,
      since it isn't a per-tile ramp.
- [x] Tectonic events: earthquakes near fault lines — new `EarthquakeManager` strikes tectonic worlds on
      the fault under the most exposed structure, damaging (and, at zero, destroying with NO refund) the
      buildings within a radius. Damage lowers `PlacedBuilding.health`, which now scales a building's
      output, and persists through save/load. Building away from the faults stays safe — the danger/reward
      trade the request describes. **NOT built:** the "locally alter elevation / spawn a volcano" aftermath
      — the surface is derived from the terrain seed and regenerates on load, so a mutated tile can't
      survive without a persistent per-tile terrain-override layer that doesn't exist yet. That is the one
      genuine follow-up (see Notes).
- [x] Drive Gas Giant/Ice/Asteroid frequency from Size — a body now rolls a MASS (0..1) first that drives
      BOTH its type (in the size-sensitive cool/cold-outer bands: the most massive become gas giants, the
      lightest asteroids) AND its actual surfaceSize, so the two finally agree. Calibrated so the mass
      cutoffs reproduce the OLD type proportions EXACTLY — this links size to type WITHOUT rebalancing
      which types appear. **Deliberately partial:** the half that would CHANGE the galaxy-wide type mix
      (a fully attribute-first pick that alters how often each type spawns) is left for a future pass — it
      needs real playtest calibration a non-compiling CI environment can't provide. `[commit: "Advanced
      Generation: Size drives type"]`

**Notes:** Built this session (Opus, at the repo owner's request to complete the whole project and file
it). Two things are intentionally NOT done, both because they need something this environment can't give:
(a) earthquake **terrain aftermath** (volcano spawn / elevation change) needs a persistent per-tile
terrain-override layer, since terrain regenerates from the seed on load; (b) the **balance-changing** half
of attribute-driven type frequency needs playtesting to keep the galaxy's world mix sane. Everything else
in §4b is built and reviewed.
